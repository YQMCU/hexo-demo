MQTT定义

MQTT是IBM开发的一个即时通讯协议。MQTT是面向M2M和物联网的连接协议，采用轻量级发布和订阅消息传输机制。Mosquitto是一款实现了MQTT v3.1协议的开源消息代理软件，提供轻量级的支持发布/订阅的消息推送模式，使设备对设备之间的短消息通信简单易用。

MQTT特点

  • MQTT协议特点–相比于RESTful架构理解RESTful架构 - 阮一峰的网络日志
    的物联网系统，MQTT协议借助消息推送功能，可以更好的实现远程控制。
  • MQTT协议角色–在RESTful架构的物联网系统，包含两个角色客户端和服务器端，而在MQTT协议中包括发布者、代理（broker，服务器）和订阅者。
  • MQTT协议消息–MQTT中的消息可以理解为发布者和订阅者交换的内容(payload，负载)，这些消息包含具体的内容，可以被订阅者使用。
  • MQTT协议主题–MQTT中的主题可理解为相同类型或相似类型的消息集合。

MQTT折腾记录(2) -- MQTT协议详解

发表于 2017-06-19

协议初解

MQTT整体上协议可以拆分为： 固定头部+可变头部+消息体。

协议说白了就是对于对方通信的一个约定，比如传过来一个字符流，第1个字节表示什么，第2个字节表示什么诸如此类的约定。

固定头部的构造如下：

bit 7 6 5 4 3 2 1 0
byte 1 [ Message Type ] DUP flag [QoS level ] RETAIN
byte 2 [ Remaining Length ]
  1. Message Type (0和15保留，共占用4个字节)
  1. DUP flag

用于保证在消息传输可靠的，如果设置为1，则在下面的变长头部里多加MessageId，并需要回复确认，保证消息传输完成，但不能用于检测消息重复发送。

  1. QoS

主要用于publish（发布消息），保证消息传递的次数：

  • 00 表示最多一次 <=1
  • 01 表示最少一次 >=1
  • 10 表示仅有一次 ==1
  • 11 保留
  1. Retain

主要用于publish（发布消息），表示服务器要保留这次推送的消息，如果有新的订阅者出现，就把这消息给它。如果不设，那么推送至当前订阅的就释放了。

  1. 固定头部的byte 2

是用来保存接下去的变长头部+消息体的总大小的。

但是不是并不是直接保存的，同样也是可以扩展的，其机制是，前7位用于保存长度，后1位用作标识。

举例：如果计算出后面的大小为0<length<=127的，正常保存，
如果计算127<length<16383的，则需要两个字节保存，将第一个字节的最大一个位置1，表示未完。然后第二个字节继续存。

无线传输电脑的声音到手机上播放

SoundWire 是一款有创意而又实用的免费虚拟声卡软件，支持将 Windows、Linux 系统以及树莓派的声音通过 WIFI 局域网无线传输串流到你的 Android 手机上播放。看起来好像很高大上，其实 SoundWire 软件的使用还是很简单的，而且也只需要求手机跟电脑是处在同一个局域网里即可。

SoundWire

当然，通过 3G/4G 的移动网络也能实现无线声音串流播放，不过你需要电脑有公网 IP 并懂得如何配置路由器让你的电脑的端口能被公网直接访问，并且还得有足够的手机流量，有具体需求的朋友可以自己研究一下。

SoundWire 使用方法教程

1、首先保证你的手机和电脑是在同一局域网内，然后在 Windows 或 Linux 系统上安装对应 SoundWire Server 服务器端软件并启动。

2、如下图可以看到，SoundWire Server 的界面并不复杂，界面上主要是显示出电脑的 IP 地址，声卡选择、音量大小调节以及将声音录音保存成 MP3 的按钮。这里我们重点是记下电脑的 IP 地址。

SoundWire Server 服务器端

特殊情况：如果你的电脑没有任何声卡，电脑上可能是没有声卡驱动的，这时你可以安装一个 Virtual Audio Cable 的虚拟声卡驱动，否则电脑本身可能无法播放任何声音，自然也就无法传输出去了。此外，XP 系统要求声卡驱动程序支持 Stereo Mix 特性，但迫于某国际唱片组织的压力，目前绝大部分声卡厂商驱动都去除了该功能，其中比较常见的 Realtek 声卡也不支持，所以建议大家还是使用 Windows 8/7 等新系统吧。

3、在 Android 手机或平板设备上安装 SoundWire 的客户端。手机版的 SoundWire 客户端分为免费版以及售价 3.99 美元的完整版。免费版在软件界面上会有广告，并且每隔 30 分钟会有一句很短的女声提示 “SoundWire Free Version”，但实际使用影响并不大。当然，如果此款软件对你很有用，建议购买正版支持一下开发商吧。

4、启动 SoundWire Android 客户端之后，APP 会自动搜索局域网内启动了 SoundWire Server 的电脑。但如果找不到，你只需手工输入电脑的 IP 地址即可连接。一旦连接成功，电脑上的服务器端软件中的 Status (状态) 会变成绿色的 Connected 字样，这时你就能在手机上听到电脑发出的所有声音了。

5、如果你在手机上听到的声音有卡顿现象，首先检查 WIFI 信号是否良好，其次可以设置 SoundWire App 里面的 Buffer 来改善。在 Menu -> Setting 里面有一个叫做 Audio Buffer Size 的设置，将其调小到 8k 可能会对声音卡顿有一定的帮助。

6、最后，是直接在手机的喇叭上播放声音，还是给手机接上音箱或耳机，就视乎你想怎么去利用 SoundWire 了。

特殊情况：如果你的电脑没有任何声卡，电脑上可能是没有声卡驱动的，这时你可以安装一个 Virtual Audio Cable 的虚拟声卡驱动，否则电脑本身可能无法播放任何声音，自然也就无法传输出去了。此外，XP 系统要求声卡驱动程序支持 Stereo Mix 特性，但迫于某国际唱片组织的压力，目前绝大部分声卡厂商驱动都去除了该功能，其中比较常见的 Realtek 声卡也不支持，所以建议大家还是使用 Windows 8/7 等新系统吧.

总结：

虽然 SoundWire 已经是一款比较老的软件了，不过其功能确实很实用。譬如对于电脑没有配备音箱却又需要听到声音的，或者想将电脑的音乐无线串流到客厅音响播放的，又或者想窝在床上看电影而耳机线又不够长的等等各种需求，SoundWire 都能满足你。

如果你也希望能将电脑的声音推送到手机上播出来，或免费打造一个“无线耳机”的话，不妨下载 SoundWire 玩玩吧。当然，你也可以再发挥下想象力看还能利用 SoundWire 做点什么有趣的事情？

MQTT折腾记录(1) -- MQTT初探

发表于 2017-06-19

MQTT(Message Queuing Telemetry Transport,消息队列遥测传输协议)，是一种基于发布/订阅(publish/subscribe)模式的轻量级通讯协议，该协议构建于TCP/IP协议上，由IBM在1999年发布。

MQTT最大的优点在于，可以以极少的代码和优先的带宽，为连接远程设备提供实时可靠的消息服务。

作为一种低开销、低带宽占用的即时通讯协议，使其在物联网、小型设备、移动应用等方面有较为广泛的应用。

MQTT协议特点

MQTT是一个基于客户端-服务器的消息发布/订阅传输协议。
MQTT协议是轻量、简单、开放和易于实现的，这些特点使它适用范围非常广泛。
在很多情况下，包括受限的环境中，如：机器与机器（M2M）和物联网（IOT）。其在，通过卫星链路通信传感器、偶尔拨号的医疗设备、智能家居及一些小型化设备中已广泛使用。

MQTT协议当前版本是2014年发布的MQTT v3.1.1.除了标准版外，还要一个简化版MQTT-SN，该协议主要针对嵌入式设备，这些设备一般工作于TCP/IP网络，如：ZigBee。

MQTT协议运行在TCP/IP或其他网络协议，提供有序、无损、双向连接。其特点包括：

  • 使用发布/订阅消息模式，它提供了一对多消息分发，以实现与应用程序的解耦
  • 对负载内容屏蔽的消息传输机制
  • 对传输消息有三种服务质量（QoS）：
    • QoS 0，最多一次，这一级别会发生消息丢失或重复，消息发布依赖于底层TCP/IP网络。即<=1
    • QoS 1，最少一次，这以级别会确保消息到达，但消息可能会重复。即>=1
    • QoS 2，恰好一次，确保消息只有一次到达，即=1.在一些要求严格的计费系统中，可以使用此级别

MQTT协议原理

MQTT协议的实现方式

  • 实现MQTT协议需要：客户端和服务器端
  • MQTT协议中有两种身份：发布者(publish)、代理(Broker)、订阅者(Subscribe)。其中消息的发表者和订阅者都是客户端，消息代理是服务器，消息发布者可以同时是订阅者。
  • MQTT传输的消息分为：主题(Topic)和负载(payload)两部分
    • Topic，可以理解为消息的类型，订阅者订阅后，就会收到该主题的消息内容(payload)
    • payload，可以理解为消息的内容，是指订阅者具体要使用的内容。

网络传输和应用消息

MQTT会构建底层网络传输：它将建立客户端到服务器的连接，提供两者之间的一个有序的、无损的、基于字节流的双向传输。
当应用数据通过MQTT网络传输时，MQTT会把与之相关的服务质量(QoS)和主题名(Topic)相关联。

MQTT客户端

一个使用MQTT协议的应用程序或者设备，它总是建立到服务器的网络连接。客户端可以：

  • 发布其他客户端会订阅的信息
  • 订阅其他客户端发布的消息
  • 退订或删除应用程序的消息
  • 断开与服务器的连接

MQTT服务器

MQTT服务器也成为消息代理(Broker)，可以是一个应用或一台设备。
它位于消息发布者和订阅者之间，它可以：

  • 接受来自客户的网络连接
  • 接收客户端发布的应用信息
  • 处理来自客户端的订阅和退订请求
  • 向订阅的客户转发应用程序消息

MQTT协议中的订阅、主题、会话

  • 订阅(Subcription)
    订阅包含主题筛选器(Topic Filter)和最大服务质量(QoS)。
    订阅会与一个会话(Session)关联。
    一个会话可以包含多个订阅。
    每一个会话中的每个订阅都有一个不同的主题筛选器。

  • 会话(Session)
    每个客户端与服务器建立连接后就是一个会话，客户端和服务器之间有状态交互。
    会话存在与一个网络之间，也可能在客户端和服务器之间跨越多个连续的网络连接。

  • 主题名(Topic Name)
    连接到一个应用程序消息的标签，该标签与服务器的订阅相匹配。
    服务器会将消息发送给订阅所匹配标签的每个客户端。

  • 主题筛选器(Topic Filter)
    一个对主题名通配符筛选，在订阅表达式中使用，表示订阅所匹配到的多个主题。

  • 负载(Payload)
    消息订阅者所具体接收到的内容

MQTT协议中的方法

MQTT协议中定义了一些方法（也被称之为动作），用来表示对确定资源所进行的操作。这个资源可以代表预先存在的数据或动态生成数据，这取决于服务器的实现。通常来说，资源指服务器上的文件或输出。

  • Connect 等待与服务器建立连接
  • Disconnect 等待MQTT客户端完成所做的工作，并与服务器断开TCP/IP会话
  • Subscribe 等待完成订阅
  • UnSubscribe 等待服务器取消客户端的一个或多个Topic订阅
  • Publish MQTT客户端发送消息请求，发送完成后返回应用程序线程

Lua模块应用程序开发架构

各个功能在各个module中处理，官方称之为库函数(lib)。自开发用户需要自己创建一个main.lua主程序，运行也是从main开始的。main.lua是一个入口文件，且只能存在一个main.lua。

脚本在sys.run()中循环运行。

所以，sys模块的run()是主架构，这个主架构是用消息机制实现的。用户拿到这个架构可以根据自己的需求增加或修改对应的消息处理程序即可实现所需的功能。

main.lua 主程序

main.lua是主程序模块，里面一般包括Require程序,初始化程序,主循环程序

run()详解

1) 该架构是用消息机制实现的

运行框架基于消息处理机制，消息来源于内部消息和外部消息。

内部消息：lua脚本调用本文件dispatch接口产生的消息，消息存储在qmsg表中。

外部消息：底层core软件产生的消息，lua脚本通过rtos.receive接口读取这些外部消息。

消息类型：

  • table型
  • Number型（使用较多）

Number类型的消息分为:

  • 定时器timeout消息(msg.id=rtos.MSG_TIMER=1)
  • 串口消息(msg.id=rtos.MSG_UART_RXDATA=2)
  • 键盘消息(msg.id=rtos.MSG_KEYPAD=3)
  • 中断消息(msg.id=rtos.MSG_INT=4)
  • 电源管理消息(msg.id=rtos.MSG_PMD=5)

补充说明：

串口消息又分为虚拟AT口(msg.uart_id=uart.ATC)和普通串口(msg.uart.id=串口号)

电源管理消息，当电池在位状态、电池电压百分比、电池电压、充电器在位状态、充电状态任何一个发生变化，就会上报该消息。

状态 变量 类型 取值范围
电池在位状态 msg.present boolean true or false
电压百分比 msg.level number 0-100
电池电压 msg.voltage number
充电器在位状态 msg.charge boolean true or false
充电器状态 msg.state number 0-不在充电 1-充电中 2-充电停止

2) 各个消息处理程序使用前需要用regmsg程序注册到handlers表中(除了定时器timeout消息和物理串口消息以外)，这样代码改动影响面小，当业务流程有修改的时候，只需要修改具体的消息处理程序的内容即可。

3) run()流程详细介绍

在ril.lua中已经用sys.regmsg(“atc”,atcreader)把ATC口的处理程序注册好了。

如果需要使用电源管理，可以单独写个应用模块，比如取名为power.lua，并在里面注册rtos.MSG_PMD消息处理程序。

如果需要GPIO中断，单独写应用模块，并在里面注册rtos.MSG_INT消息的处理程序，并定义相应的处理程序。

3) 在被调用模块加载后，可以在调用模块中使用被调模块的全局变量和全局程序。

程序注册

使用相关程序做各种处理的时候，需要先将程序注册下，否则无法使用。

这样做的好处是可以将同类型程序集中放置在一个表中，方便管理和修改。

除了以上提及regmsg()外，还有regurc()，regapp()，regnotify()，reguart()。

regmsg()用来注册相应的消息处理程序

regmsg()位于sys.lua库文件中。luat主架构中的sys.run()目前能够处理的消息中，除了定时器消息（定时器函数直接用sys.timer_start来启动）和物理串口消息（用reguart()来注册）外，其余内部消息，例如AT命令的虚拟串口数据接收消息、音频消息、充电管理消息、按键消息等，均需要用regmsg注册相应的消息处理函数。

  • 串口消息
    又分为虚拟AT口(msg.uart_id=uart.ATC)和物理串口(msg.uart_id=串口号)消息。
    其中虚拟AT口的消息已经缺省注册了，不需要用户再注册；
    物理串口消息处理函数使用前需要用reguart()注册。

  • 键盘消息
    msg.id=rtos.MSG_KEYPAD=3
    需要用regmsg()来注册处理函数

  • 中断消息
    msg.id=rtos.MSG_INT=4
    需要用regmsg()来注册处理函数

  • 电源管理消息
    msg.id=rtos.MSG_PMD=5
    需要用regmsg()来注册处理函数

  • 音频消息
    msg.id=rtos.MSG_AUDIO
    需要用regmsg()来注册处理函数

regmsg()函数

regmsg()处理流程如下：

1) 在脚本某处regmsg(msg.id,proc),将消息处理函数proc的地址注册在handlers表中，表的key是msg.id

2) 主循环程序run()中，当rtos.receive()收到某个消息msg时，自动执行该消息的msg.id对应的处理程序。

regapp()用来注册应用程序

对AT命令的查询结果或者某些事件的处理结果，会以dispatch(消息id,参数1,参数2,…)的形式来通知，如果想在app中对该消息id及对应的参数(参数1,参数2,…)进行进一步处理，需要事先以regapp方式注册该app程序，注册后，每次dispatch相应的消息id，就自动会进入相应的app程序对dispatch出来的消息id和参数进行处理。

regapp的详细流程如下：

1) 在脚本某处执行regapp()，即dispatch(“SYS_ADD_APP”,app)

2) dispatch函数将(“SYS_ADD_APP”,app)插入表qmsg尾部

3) 在下一轮的run()中，执行runqmsg()->getmsg()，获取到(“SYS_ADD_APP”,app)

4) 执行callapp(“SYS_ADD_APP”,app)，将app这个回调函数的地址指针放到apps这个表的尾部

regapp有2种注册方式:

a) 以表的形式来注册

regrsp()用来注册AT命令处理程序

当用户发送AT命令时，一般是需要这些命令的返回结果的，
AT命令返回结果的处理程序app是通过regrsp()函数来注册的在rsptable表中的。
这样当发送AT命令时，会自动执行该处理程序处理AT命令返回结果。

对AT命令的返回结果的处理，一般有两种情况：
1) 一些常用的AT命令的处理程序，库文件已经缺省用regrsp注册了。此时处理程序会把AT命令结果dispatch出来，后续再用regapp注册的app再做进一步处理（适用于异步处理或前后有因果关系的情况）
2) 如果AT命令在lib库文件中没有被regrsp注册过，则需要用regrsp()注册相应的处理程序。
此时用户可以在这些处理程序中直接处理AT命令返回结果（适用于同步处理），也可以用1的异步处理方式。

regrsp 处理流程如下：

1) 在脚本某处regrsp(head,fnc,typ)，将处理函数func的地址注册在rsptable表中，表的key是head(AT命令返回结果的前缀，比如+CSQ)

2) 主循环程序run()中，当rtos.receive()收到某个AT命令返回结果时，在atcreader()->procatc()中自动执行该AT对应的处理程序
