Lua模块应用程序开发架构

各个功能在各个module中处理，官方称之为库函数(lib)。自开发用户需要自己创建一个main.lua主程序，运行也是从main开始的。main.lua是一个入口文件，且只能存在一个main.lua。

脚本在sys.run()中循环运行。

所以，sys模块的run()是主架构，这个主架构是用消息机制实现的。用户拿到这个架构可以根据自己的需求增加或修改对应的消息处理程序即可实现所需的功能。

main.lua 主程序

main.lua是主程序模块，里面一般包括Require程序,初始化程序,主循环程序

run()详解

1) 该架构是用消息机制实现的

运行框架基于消息处理机制，消息来源于内部消息和外部消息。

内部消息：lua脚本调用本文件dispatch接口产生的消息，消息存储在qmsg表中。

外部消息：底层core软件产生的消息，lua脚本通过rtos.receive接口读取这些外部消息。

消息类型：

  • table型
  • Number型（使用较多）

Number类型的消息分为:

  • 定时器timeout消息(msg.id=rtos.MSG_TIMER=1)
  • 串口消息(msg.id=rtos.MSG_UART_RXDATA=2)
  • 键盘消息(msg.id=rtos.MSG_KEYPAD=3)
  • 中断消息(msg.id=rtos.MSG_INT=4)
  • 电源管理消息(msg.id=rtos.MSG_PMD=5)

补充说明：

串口消息又分为虚拟AT口(msg.uart_id=uart.ATC)和普通串口(msg.uart.id=串口号)

电源管理消息，当电池在位状态、电池电压百分比、电池电压、充电器在位状态、充电状态任何一个发生变化，就会上报该消息。

状态 变量 类型 取值范围
电池在位状态 msg.present boolean true or false
电压百分比 msg.level number 0-100
电池电压 msg.voltage number
充电器在位状态 msg.charge boolean true or false
充电器状态 msg.state number 0-不在充电 1-充电中 2-充电停止

2) 各个消息处理程序使用前需要用regmsg程序注册到handlers表中(除了定时器timeout消息和物理串口消息以外)，这样代码改动影响面小，当业务流程有修改的时候，只需要修改具体的消息处理程序的内容即可。

3) run()流程详细介绍

在ril.lua中已经用sys.regmsg(“atc”,atcreader)把ATC口的处理程序注册好了。

如果需要使用电源管理，可以单独写个应用模块，比如取名为power.lua，并在里面注册rtos.MSG_PMD消息处理程序。

如果需要GPIO中断，单独写应用模块，并在里面注册rtos.MSG_INT消息的处理程序，并定义相应的处理程序。

模块的调用关系

1) 需要使用这些模块的时候，只需要在调用模块中require加载该模块即可。

2) 调用模块可以是main.lua，也可以是各个分模块。但是被调用模块不可以是main.lua。

3) 在被调用模块加载后，可以在调用模块中使用被调模块的全局变量和全局程序。

程序注册

使用相关程序做各种处理的时候，需要先将程序注册下，否则无法使用。

这样做的好处是可以将同类型程序集中放置在一个表中，方便管理和修改。

除了以上提及regmsg()外，还有regurc()，regapp()，regnotify()，reguart()。

regmsg()用来注册相应的消息处理程序

regmsg()位于sys.lua库文件中。luat主架构中的sys.run()目前能够处理的消息中，除了定时器消息（定时器函数直接用sys.timer_start来启动）和物理串口消息（用reguart()来注册）外，其余内部消息，例如AT命令的虚拟串口数据接收消息、音频消息、充电管理消息、按键消息等，均需要用regmsg注册相应的消息处理函数。

  • 串口消息
    又分为虚拟AT口(msg.uart_id=uart.ATC)和物理串口(msg.uart_id=串口号)消息。
    其中虚拟AT口的消息已经缺省注册了，不需要用户再注册；
    物理串口消息处理函数使用前需要用reguart()注册。

  • 键盘消息
    msg.id=rtos.MSG_KEYPAD=3
    需要用regmsg()来注册处理函数

  • 中断消息
    msg.id=rtos.MSG_INT=4
    需要用regmsg()来注册处理函数

  • 电源管理消息
    msg.id=rtos.MSG_PMD=5
    需要用regmsg()来注册处理函数

  • 音频消息
    msg.id=rtos.MSG_AUDIO
    需要用regmsg()来注册处理函数

regmsg()函数

regmsg()处理流程如下：

1) 在脚本某处regmsg(msg.id,proc),将消息处理函数proc的地址注册在handlers表中，表的key是msg.id

2) 主循环程序run()中，当rtos.receive()收到某个消息msg时，自动执行该消息的msg.id对应的处理程序。

regapp()用来注册应用程序

对AT命令的查询结果或者某些事件的处理结果，会以dispatch(消息id,参数1,参数2,…)的形式来通知，如果想在app中对该消息id及对应的参数(参数1,参数2,…)进行进一步处理，需要事先以regapp方式注册该app程序，注册后，每次dispatch相应的消息id，就自动会进入相应的app程序对dispatch出来的消息id和参数进行处理。

regapp的详细流程如下：

1) 在脚本某处执行regapp()，即dispatch(“SYS_ADD_APP”,app)

2) dispatch函数将(“SYS_ADD_APP”,app)插入表qmsg尾部

3) 在下一轮的run()中，执行runqmsg()->getmsg()，获取到(“SYS_ADD_APP”,app)

4) 执行callapp(“SYS_ADD_APP”,app)，将app这个回调函数的地址指针放到apps这个表的尾部

regapp有2种注册方式:

a) 以表的形式来注册

regrsp()用来注册AT命令处理程序

当用户发送AT命令时，一般是需要这些命令的返回结果的，
AT命令返回结果的处理程序app是通过regrsp()函数来注册的在rsptable表中的。
这样当发送AT命令时，会自动执行该处理程序处理AT命令返回结果。

对AT命令的返回结果的处理，一般有两种情况：
1) 一些常用的AT命令的处理程序，库文件已经缺省用regrsp注册了。此时处理程序会把AT命令结果dispatch出来，后续再用regapp注册的app再做进一步处理（适用于异步处理或前后有因果关系的情况）
2) 如果AT命令在lib库文件中没有被regrsp注册过，则需要用regrsp()注册相应的处理程序。
此时用户可以在这些处理程序中直接处理AT命令返回结果（适用于同步处理），也可以用1的异步处理方式。

regrsp 处理流程如下：

1) 在脚本某处regrsp(head,fnc,typ)，将处理函数func的地址注册在rsptable表中，表的key是head(AT命令返回结果的前缀，比如+CSQ)

2) 主循环程序run()中，当rtos.receive()收到某个AT命令返回结果时，在atcreader()->procatc()中自动执行该AT对应的处理程序
